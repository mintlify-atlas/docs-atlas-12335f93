## Overview

The `configRepo` section defines how VCVerifier manages service configurations, credential types, and trust relationships. This can be configured statically in YAML or dynamically via a Credentials Config Service.

## ConfigRepo Settings

<ParamField path="configRepo.configEndpoint" type="string">
  URL of the Credentials Config Service for dynamic configuration updates.
  
  When configured, VCVerifier periodically fetches service configurations from this endpoint.
  
  ```yaml
  configRepo:
    configEndpoint: http://config-service:8080
  ```
</ParamField>

<ParamField path="configRepo.updateInterval" type="integer" default="30">
  Interval in seconds for fetching configuration updates from the config service.
  
  ```yaml
  configRepo:
    configEndpoint: http://config-service:8080
    updateInterval: 60
  ```
</ParamField>

<Note>
  If `configEndpoint` is not configured, only static configuration from the YAML file will be used.
</Note>

## Service Configuration

<ParamField path="configRepo.services" type="array">
  Array of statically configured services with their trust anchors and scopes.
</ParamField>

Each service has the following structure:

### Service Properties

<ParamField path="services[].id" type="string" required>
  Unique identifier for the service.
  
  ```yaml
  configRepo:
    services:
      - id: myService
  ```
</ParamField>

<ParamField path="services[].defaultOidcScope" type="string">
  Default OIDC scope to use when none is specified in the authentication request.
  
  ```yaml
  configRepo:
    services:
      - id: myService
        defaultOidcScope: "default"
  ```
</ParamField>

<ParamField path="services[].authorizationType" type="string">
  Type of authorization to be provided in the redirect response.
</ParamField>

<ParamField path="services[].authorizationPath" type="string">
  Path for the authorization endpoint to be provided in redirects.
</ParamField>

## OIDC Scopes

<ParamField path="services[].oidcScopes" type="object" required>
  Map of scope names to scope configurations. Each scope defines which credentials to request and how to trust them.
</ParamField>

### Scope Configuration

<ParamField path="oidcScopes.<scopeName>.credentials" type="array" required>
  Array of credential configurations for this scope.
</ParamField>

<ParamField path="oidcScopes.<scopeName>.presentationDefinition" type="object">
  Presentation Definition following the [DIF Presentation Exchange](https://identity.foundation/presentation-exchange/) specification.
  
  Defines the format and constraints for requested credentials.
</ParamField>

<ParamField path="oidcScopes.<scopeName>.dcql" type="object">
  DC Query Language (DCQL) query for requesting specific credentials and claims.
</ParamField>

<ParamField path="oidcScopes.<scopeName>.flatClaims" type="boolean" default="false">
  When `true`, flattens credential claims to plain JWT claims instead of keeping the credential/presentation structure.
  
  ```yaml
  oidcScopes:
    myScope:
      flatClaims: true
  ```
</ParamField>

## Credential Configuration

Each credential in a scope has the following properties:

<ParamField path="credentials[].type" type="string" required>
  Type of the credential (e.g., `VerifiableCredential`, `CustomerCredential`).
  
  ```yaml
  credentials:
    - type: CustomerCredential
  ```
</ParamField>

### Trust Configuration

<ParamField path="credentials[].trustedParticipantsLists" type="array">
  List of trusted participants registries to check if the credential issuer is a trusted participant.
  
  Each entry has:
  - `type` - Registry type (`ebsi` or `gaia-x`)
  - `url` - URL of the registry
  
  ```yaml
  credentials:
    - type: CustomerCredential
      trustedParticipantsLists:
        - type: ebsi
          url: https://tir-pdc.ebsi.fiware.dev
        - type: gaia-x
          url: https://registry.lab.gaia-x.eu
  ```
</ParamField>

<ParamField path="credentials[].trustedIssuersLists" type="string[]">
  List of EBSI Trusted Issuers Registry compatible endpoints to verify that issuers are trusted and have permission to issue credentials with specific claims.
  
  ```yaml
  credentials:
    - type: CustomerCredential
      trustedIssuersLists:
        - https://tir-pdc.ebsi.fiware.dev
  ```
</ParamField>

<Accordion title="EBSI vs Gaia-X Registries">
  **EBSI Trusted Issuers Registry**:
  - Standard EBSI-compliant registry
  - Default type for backwards compatibility
  - Supports any DID method
  
  **Gaia-X Registry**:
  - Gaia-X Digital Clearing House registry
  - Requires `did:web` identifiers
  - Issuers must provide valid `x5u` location in their `publicKeyJwk`
  
  You can configure multiple registries of different types. An issuer is trusted if found in at least one registry.
</Accordion>

### Holder Verification

<ParamField path="credentials[].holderVerification.enabled" type="boolean" default="false">
  When `true`, verifies that the credential holder matches the presenter.
  
  ```yaml
  credentials:
    - type: CustomerCredential
      holderVerification:
        enabled: true
        claim: subject
  ```
</ParamField>

<ParamField path="credentials[].holderVerification.claim" type="string">
  Claim path to retrieve the holder identifier from the credential.
</ParamField>

### Compliance Verification

<ParamField path="credentials[].requireCompliance" type="boolean" default="false">
  When `true`, requires a compliance credential accompanying this credential.
  
  ```yaml
  credentials:
    - type: ParticipantCredential
      requireCompliance: true
  ```
</ParamField>

### JWT Inclusion

<ParamField path="credentials[].jwtInclusion.enabled" type="boolean" default="true">
  Whether to include this credential in the generated JWT.
</ParamField>

<ParamField path="credentials[].jwtInclusion.fullInclusion" type="boolean" default="false">
  When `true`, embeds the complete credential. When `false`, only configured claims are included.
</ParamField>

<ParamField path="credentials[].jwtInclusion.claimsToInclude" type="array">
  Array of claims to include from the credential. Each entry has:
  - `originalKey` - Key path in the credential
  - `newKey` - Key to use in the JWT (optional, defaults to originalKey)
  
  ```yaml
  credentials:
    - type: CustomerCredential
      jwtInclusion:
        enabled: true
        fullInclusion: false
        claimsToInclude:
          - originalKey: credentialSubject.email
            newKey: email
          - originalKey: credentialSubject.name
  ```
</ParamField>

## Presentation Definition

The presentation definition follows the [DIF Presentation Exchange](https://identity.foundation/presentation-exchange/) specification:

<CodeGroup>
```yaml Presentation Definition
oidcScopes:
  default:
    presentationDefinition:
      id: my-presentation
      input_descriptors:
        - id: my-descriptor
          constraints:
            fields:
              - id: credential-type
                path:
                  - $.type
                filter:
                  type: array
                  contains:
                    const: CustomerCredential
          format:
            jwt_vc:
              alg:
                - ES256
```

```yaml SD-JWT Format
oidcScopes:
  default:
    presentationDefinition:
      id: my-presentation
      input_descriptors:
        - id: my-descriptor
          constraints:
            fields:
              - id: vct-field
                path:
                  - $.vct
                filter:
                  const: "CustomerCredential"
          format:
            'sd+jwt-vc':
              alg:
                - ES256
```
</CodeGroup>

### Input Descriptor

<ParamField path="input_descriptors[].id" type="string" required>
  Unique identifier for the input descriptor.
</ParamField>

<ParamField path="input_descriptors[].constraints.fields" type="array">
  Array of field constraints:
  - `id` - Field identifier
  - `path` - JSONPath array to the field
  - `optional` - Whether the field is optional (default: `true`)
  - `filter` - Custom filter to apply (e.g., value constraints)
</ParamField>

<ParamField path="input_descriptors[].format" type="object">
  Credential format requirements. Common formats:
  - `jwt_vc` - JWT Verifiable Credential
  - `ldp_vc` - JSON-LD Verifiable Credential
  - `sd+jwt-vc` - Selective Disclosure JWT VC
  
  Each format can specify `alg` (algorithms) requirements.
</ParamField>

## DCQL Query

DC Query Language provides fine-grained control over requested credentials:

```yaml
oidcScopes:
  default:
    dcql:
      credentials:
        - id: customer-credential
          format: jwt_vc
          claims:
            - path:
                - $.vc.credentialSubject.email
          meta:
            vct_values:
              - CustomerCredential
          trusted_authorities:
            - type: trust_registry
              values:
                - https://tir-pdc.ebsi.fiware.dev
```

<ParamField path="dcql.credentials" type="array" required>
  Array of credential queries.
</ParamField>

<ParamField path="dcql.credential_sets" type="array">
  Array of credential set queries for complex requirements.
</ParamField>

## Complete Example

```yaml
configRepo:
  # Optional: Use external config service
  configEndpoint: http://config-service:8080
  updateInterval: 60
  
  # Static service configuration
  services:
    - id: marketplace
      defaultOidcScope: "customer"
      
      oidcScopes:
        customer:
          # Credentials and trust configuration
          credentials:
            - type: CustomerCredential
              trustedParticipantsLists:
                - type: ebsi
                  url: https://tir-pdc.ebsi.fiware.dev
              trustedIssuersLists:
                - https://tir-pdc.ebsi.fiware.dev
              holderVerification:
                enabled: true
                claim: subject
              requireCompliance: false
              jwtInclusion:
                enabled: true
                fullInclusion: false
                claimsToInclude:
                  - originalKey: credentialSubject.email
                    newKey: email
                  - originalKey: credentialSubject.customerId
          
          # Presentation definition
          presentationDefinition:
            id: customer-presentation
            input_descriptors:
              - id: customer-credential-descriptor
                constraints:
                  fields:
                    - id: credential-type
                      path:
                        - $.type
                      filter:
                        type: array
                        contains:
                          const: CustomerCredential
                format:
                  jwt_vc:
                    alg:
                      - ES256
          
          # Flatten claims in JWT
          flatClaims: true
        
        premium:
          credentials:
            - type: PremiumCustomerCredential
              trustedParticipantsLists:
                - type: ebsi
                  url: https://tir-pdc.ebsi.fiware.dev
              jwtInclusion:
                enabled: true
                fullInclusion: true
```

## Best Practices

<Warning>
  **Trust Configuration**:
  - Always configure at least one trusted participants or issuers list
  - Use multiple registries for redundancy
  - Regularly review and update trust anchors
</Warning>

<Note>
  **Scope Design**:
  - Create separate scopes for different credential requirements
  - Use descriptive scope names that reflect the credential type
  - Configure `defaultOidcScope` for seamless user experience
</Note>

<Accordion title="Dynamic vs Static Configuration">
  **When to use static configuration**:
  - Simple deployments with few services
  - Fixed trust relationships
  - Development and testing
  
  **When to use dynamic configuration**:
  - Multiple services with changing requirements
  - Need for runtime configuration updates
  - Production environments with evolving trust frameworks
  - Multi-tenant scenarios
</Accordion>
